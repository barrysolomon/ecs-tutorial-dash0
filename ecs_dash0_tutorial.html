Dash0 on AWS ECS

Logs + Distributed Tracing — Step-by-Step Setup
Practice guide  •  Fargate & EC2 launch types
ECS Fargate OpenTelemetry FireLens

What you're setting up

This guide walks through getting full observability on an ECS-hosted application: distributed traces (via OTel SDK) and logs (via FireLens / Fluent Bit), both flowing into Dash0 with trace-log correlation. It is written for Fargate but notes where EC2 differs.

Architecture at a glance:

Your App Container
OTLP gRPC :4317
OTel Collector Sidecar
Fluent BitFireLens log router
OTLP / gRPC
Dash0 Ingest Endpointapi.<region>.aws.dash0.com:4317
Tip
You can also export directly from the app to Dash0 (no collector sidecar). That is the simplest starting point if the customer just wants traces fast. Use the sidecar when they need log enrichment, PII scrubbing, or sampling control.

Prerequisites

  • AWS account with ECS cluster (Fargate or EC2 launch type)
  • Docker image for a service you can redeploy
  • Dash0 trial org — see Step 1
  • AWS CLI configured locally (aws sts get-caller-identity should work)
  • Basic familiarity with ECS task definitions (JSON or console)
1

Create your Dash0 org and get an auth token

5 minutes — do this first, everything else depends on it

1a — Sign up / log in

  1. Go to app.dash0.com and sign up or log in
  2. Create a new Organisation (or use an existing trial org)
  3. Note the Organisation slug — you will need it for the OTLP endpoint URL

1b — Create an auth token

  1. Organisation Settings → Auth Tokens → New Token
  2. Name it something obvious: ecs-onboarding
  3. Select All permissions (or at minimum Ingest)
  4. Copy the token — it is only shown once
Tip
Store the token in AWS Secrets Manager right now. You will reference it from the task definition as a secret, not a plain env var.

1c — Store token in Secrets Manager

aws secretsmanager create-secret \
  --name dash0/auth-token \
  --secret-string 'auth_xxxxxxxxxxxxxxxxxxxx'

1d — Know your OTLP endpoint

Dash0 uses regional OTLP endpoints. For most EU/UK customers:

EU (Ireland)
api.eu-west-1.aws.dash0.com:4317
US West
api.us-west-2.aws.dash0.com:4317
US East
api.us-east-2.aws.dash0.com:4317
2

Instrument your application

Add the OTel SDK — Node.js, Python, Java or Go

Option A — Auto-instrumentation (Node.js)

If the app is Node.js you can get traces with zero code changes using the OTel auto-instrumentation package:

npm install @opentelemetry/sdk-node \
            @opentelemetry/auto-instrumentations-node \
            @opentelemetry/exporter-trace-otlp-grpc

Create tracing.js (loaded before anything else):

const { NodeSDK } = require('@opentelemetry/sdk-node');
const { getNodeAutoInstrumentations } = require('@opentelemetry/auto-instrumentations-node');
const { OTLPTraceExporter } = require('@opentelemetry/exporter-trace-otlp-grpc');

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter(),  // reads OTEL_EXPORTER_OTLP_ENDPOINT
  instrumentations: [getNodeAutoInstrumentations()]
});
sdk.start();

Start your app with:

node --require ./tracing.js server.js
Other languages
For Python use opentelemetry-distro + opentelemetry-bootstrap. For Java use the -javaagent flag with the OTel Java agent JAR. For Go, instrument manually using go.opentelemetry.io/otel.

Option B — Direct OTLP export (no collector sidecar)

Skip the sidecar entirely — set these env vars in the task definition and the app will send directly to Dash0:

VariableValue
OTEL_EXPORTER_OTLP_ENDPOINThttps://api.eu-west-1.aws.dash0.com:4317
OTEL_EXPORTER_OTLP_HEADERSAuthorization=Bearer <your-token>
OTEL_SERVICE_NAMEyour-service-name
OTEL_RESOURCE_ATTRIBUTESdeployment.environment=production,team=platform
Tip
Direct export is the right starting point. Move to a sidecar collector when you need sampling, PII redaction, or want to decouple telemetry config from app deploys.
3

Add OTel Collector sidecar to your task definition

Skip if using Option B (direct export) above

3a — Collector config file

Store this as an SSM parameter or bake into a config image. Save as otel-collector-config.yaml:

receivers:
  otlp:
    protocols:
      grpc:          # listens on :4317
      http:          # listens on :4318

processors:
  batch:             # buffer + batch before sending
    timeout: 5s
    send_batch_size: 1000
  resource:
    attributes:      # add cluster metadata
      - action: insert
        key: cloud.provider
        value: aws
      - action: insert
        key: cloud.platform
        value: aws_ecs

exporters:
  otlp/dash0:
    endpoint: api.eu-west-1.aws.dash0.com:4317
    headers:
      Authorization: 'Bearer ${DASH0_AUTH_TOKEN}'

service:
  pipelines:
    traces:
      receivers:  [otlp]
      processors: [resource, batch]
      exporters:  [otlp/dash0]
    logs:
      receivers:  [otlp]
      processors: [resource, batch]
      exporters:  [otlp/dash0]

3b — Task definition — container definitions

Add the collector as a sidecar alongside your app container:

{
  "containerDefinitions": [
    {
      "name": "app",
      "image": "your-ecr-image:latest",
      "environment": [
        { "name": "OTEL_EXPORTER_OTLP_ENDPOINT", "value": "http://localhost:4317" },
        { "name": "OTEL_SERVICE_NAME",            "value": "my-service" },
        { "name": "OTEL_RESOURCE_ATTRIBUTES",     "value": "deployment.environment=prod" }
      ]
    },
    {
      "name": "otel-collector",
      "image": "otel/opentelemetry-collector-contrib:latest",
      "essential": false,
      "command": ["--config=/etc/otel/config.yaml"],
      "secrets": [
        { "name": "DASH0_AUTH_TOKEN",
          "valueFrom": "arn:aws:secretsmanager:eu-west-1:ACCOUNT:secret:dash0/auth-token" }
      ],
      "portMappings": [
        { "containerPort": 4317, "protocol": "tcp" },
        { "containerPort": 4318, "protocol": "tcp" }
      ],
      "mountPoints": [
        { "sourceVolume": "otel-config", "containerPath": "/etc/otel" }
      ]
    }
  ]
}
Tip
In Fargate, all containers in a task share a network namespace. localhost:4317 in the app container reaches the collector sidecar directly — no service discovery needed.
4

Set up log collection with FireLens + Fluent Bit

How ECS captures stdout logs and routes them to Dash0

Why FireLens

ECS Fargate does not have a node-level log agent like K8s DaemonSet. FireLens is AWS's log router — it wraps Fluent Bit as a sidecar and intercepts all container stdout/stderr. You configure it to forward to the OTel Collector (or directly to Dash0 via OTLP).

4a — Add FireLens container to task definition

{
  "name": "log_router",
  "image": "906394416424.dkr.ecr.eu-west-1.amazonaws.com/aws-for-fluent-bit:stable",
  "essential": true,
  "firelensConfiguration": {
    "type": "fluentbit",
    "options": { "enable-ecs-log-metadata": "true" }
  }
}

4b — Point app container logs at FireLens

In your app container definition, change logConfiguration to use the awsfirelens driver:

"logConfiguration": {
  "logDriver": "awsfirelens",
  "options": {
    "Name":   "forward",
    "Host":   "localhost",
    "Port":   "24224",
    "Match":  "*"
  }
}

4c — Fluent Bit output to OTel Collector

Forward to the OTel Collector sidecar, which then sends to Dash0:

[OUTPUT]
    Name          opentelemetry
    Match         *
    Host          127.0.0.1
    Port          4318
    logs_uri      /v1/logs
    tls           off
    header        Authorization Bearer ${DASH0_AUTH_TOKEN}
Tip
Fluent Bit's opentelemetry output plugin (available in aws-for-fluent-bit:stable) converts log records to OTLP format before sending. This is cleaner than using the HTTP output plugin.
5

Wire up trace-log correlation

Link every log line back to its parent trace

For logs to link to traces in Dash0, each log line needs trace_id and span_id fields that match the active OTel span. There are two approaches:

Option A — Inject trace context in the app (recommended)

Use the OTel logging bridge in your logging library. Example for Node.js with Winston:

const { context, trace } = require('@opentelemetry/api');
const winston = require('winston');

const logger = winston.createLogger({
  format: winston.format.combine(
    winston.format((info) => {
      const span = trace.getActiveSpan();
      if (span) {
        const ctx = span.spanContext();
        info.trace_id = ctx.traceId;
        info.span_id  = ctx.spanId;
      }
      return info;
    })(),
    winston.format.json()
  ),
  transports: [new winston.transports.Console()]
});

Every log line emitted inside a traced request will contain trace_id and span_id as JSON fields. Dash0 auto-correlates these.

Option B — OTel log appender (SDK-level)

If using the OTel SDK log bridge, configure it as an additional log exporter in your SDK setup. This sends logs directly via OTLP with trace context already attached — no manual injection needed. Supported in Java (log4j2/logback), Python (logging bridge), and Node.js (experimental).

Tip
Both options produce the same result in Dash0. Option A is more universally supported today across frameworks. Option B is cleaner long-term.
6

Deploy and verify

Confirm data is flowing into Dash0

6a — Deploy the updated task definition

# Register new task definition revision
aws ecs register-task-definition --cli-input-json file://task-definition.json

# Update the service to use the new revision
aws ecs update-service \
  --cluster your-cluster \
  --service your-service \
  --task-definition your-task-family:REVISION

6b — Check collector health

# Exec into the collector sidecar
aws ecs execute-command \
  --cluster your-cluster \
  --task TASK_ID \
  --container otel-collector \
  --interactive \
  --command '/bin/sh'

# Inside the container:
curl http://localhost:13133/  # health check endpoint

6c — Verify in Dash0

  1. Open app.dash0.com → your organisation
  2. Go to Services — you should see your service within ~60 seconds of first traffic
  3. Click into the service → Traces tab to see spans
  4. Logs tab — verify log records are appearing
  5. Click a trace → check for correlated log entries in the side panel

6d — Common issues

SymptomFix
No data in Dash0 Check collector logs for auth errors. Token may be wrong or expired. Verify DASH0_AUTH_TOKEN secret is accessible to the task role.
Traces but no logs Check FireLens is configured and the app logDriver is awsfirelens, not awslogs. Verify Fluent Bit output is pointed at correct port.
Logs but no trace correlation App is not injecting trace_id / span_id into log records. Add the OTel trace context injection (Step 5).
Collector exits immediately Config YAML syntax error. Check CloudWatch logs for the otel-collector container. Common: missing service.pipelines block.
403 from Dash0 endpoint Auth token lacks Ingest permission, or wrong token format. Should be Bearer auth_xxxxxxx in the Authorization header.

Demo walkthrough

Suggested order for a live demo:

  1. Show the service appearing in the Dash0 service catalog (RED metrics — rate, errors, duration)
  2. Drill into a trace — show the waterfall, span attributes, parent-child relationships
  3. From the trace, click into correlated logs — the key feature for ECS observability
  4. Show a log entry with trace_id / span_id and click it to jump back to the trace
  5. Pull up the Dash0 Logs view, filter by service name, search across all log fields
Framing
For teams migrating to ECS (from Elastic Beanstalk, EC2, or Lambda): "you get full observability from day one of the migration, no bolt-on required."
